## 1. 当前项目是什么

- 这是一个 Web-first 的智能运维工作台。
- 前端主骨架已稳定。
- 当前重点是补后端能力，不再是桌面迁移。

## 2. 当前代码主骨架

- `config/appConfig.js`：前后端共享本地地址配置
- `src/components/`：界面
- `src/components/Overview/`：总览工作区，负责展示型运行总览，不承载复杂配置
- `src/services/contracts.ts`：前后端共享契约
- `src/services/runtime/`：前端系统能力入口
- `src/stores/index.ts`：应用状态中枢
- `server/`：本地后端桥接
- `skills/`：正式 Skill 定义
- `scripts/`：历史脚本样例 / 后续执行模板来源
- `scripts/dev-all.js`：本地一键同时启动前后端
- `npm run dev:server`：当前用普通 `node` 启动，避免本机 `node --watch` 触发 `EMFILE`

## 3. 改代码时必须遵守

1. 系统能力先改 `contracts.ts` 和 `runtime/*`，不要在组件里直接堆接口细节。
2. 本地地址、端口、代理目标先看 `config/appConfig.js` 和 `.env`，不要在前后端各写一份硬编码。
3. 不要恢复 Tauri、本地 Python 路径、本地目录提示这类桌面端心智。
4. 不要新增第二套设置、工具、脚本面板。
5. 新功能优先落在现有主骨架里，不要绕开 `src/components/*` / `server/*` 再长旁路。
6. 模型提供商展示遵循当前产品约定：国内厂商优先，`OpenAI` 文案保持简洁统一，不再使用 `OpenAI 兼容`。
7. Skill 命名和脚本颗粒度对齐：`skill.name`、Skill 目录名、脚本 basename 保持一致（不带 `.py`）。
8. `总览` 工作区是展示页，不要在里面再长一套任务操作后台；复杂启停和配置仍留在任务工作台。
9. `总览` 首屏优先展示任务态势，不把“后端在线”这类基础连接状态做成主视觉卡片。
10. `总览` 避免重复统计和说明性堆叠；首屏只承载状态摘要、当前状态和最近事件，不再做第二套数字面板。
11. `总览` 的摘要卡要保持紧凑和可扫读，避免出现“大面积留白 + 数字悬空”的空心卡片。
12. `总览` 摘要图标优先使用语义直接、轮廓稳定的图标，避免使用辨识度弱或视觉重心飘的符号。
13. `总览` 指标卡中的图标容器和 SVG 本体都要显式校正尺寸与居中，避免出现“视觉上偏上/偏下”的假居中。
14. `总览` 健康摘要的四张指标卡统一采用“左图标 + 右标签与数值”的横向布局，不再使用图标和数值上下分离的堆叠结构。
15. 这类摘要指标优先用稳定的 grid 双列布局，不要依赖 flex 的内容高度去做“看起来差不多”的居中。
16. 不要用 `.overview-summary-tile span` 这类宽泛选择器给摘要卡通用上样式，图标容器、标签和数值必须用专用类分开控制。
17. `总览` 首屏职责固定为“健康摘要 + 活跃告警 + 最近事件流”；健康任务不要再重复出现在活跃告警区，时间线也不要承担当前状态摘要。
18. `总览` 允许在前端对 10 分钟窗口内的重复告警做折叠聚合，但不要因此改写原始任务状态；目标展示优先用结构化 target，其次从事件详情提取，最后才回退到“目标未标注”。
19. `Skills` 标签页顶部避免堆大块说明卡；上传区保留稳定盒子尺寸，说明收进提示图标，浮层内容必须完整显示，不要被裁切。
20. 顶部轻工具条中的标题、提示图标和按钮要用局部样式锁定垂直基线，不要只依赖通用 `.btn` 规则去“碰运气”居中。
21. `MCP Servers` 标签页优先展示收窄后的 Server 摘要卡；名字和状态标签分成清晰两层，展开后再做编辑。
22. 侧边栏的展开/收缩动画必须采用“布局壳补位 + 面板滑动 + 内容淡出”的模式，不允许再用宽度直接挤压文字和列表内容。
23. 侧边栏收起后的恢复入口统一保留在顶部栏，不再额外渲染侧栏边缘展开把手，避免重复控制入口和叠影。
24. 任务上传入口优先挂在“即时任务 / 托管任务”这类任务类型行的右侧，先做轻量图标按钮，不要把上传入口塞到全局工具栏或设置里。
25. 页面刷新后的连续性默认要保留当前工作区和当前对话；弹层、瞬时聚焦和其他临时 UI 状态不要默认持久化。
26. 脚本上传 v1 先只落 `scripts/instant/` 或 `scripts/managed/`，并把一句说明写入旁路 `.meta.json`；Skill 壳生成函数可保留，但这轮不要自动生成或自动接入任务体系。
27. 任务上传弹层优先贴近业务面板风格：灰色遮罩、白色卡面、轻边框、自定义文件选择区和柔和进入动画，不直接暴露原生文件输入控件。
28. 任务上传弹层标题统一使用“上传脚本”；文件选择只允许点击“选择文件”动作按钮触发，且支持 `Esc` 关闭。
29. 主题和背景首屏要避免刷新闪烁；如果外观状态来自本地存储，优先在 `index.html` 启动阶段提前写入 `data-theme` / `data-bg` 和基础背景色，不要等应用挂载后再切换。
30. 刷新恢复链路分三层：`index.html` 只负责首帧外观 bootstrap；`persistence.ts` 中 bootstrap 只读本地 UI 快照，正式配置合并留给运行时；`initializeStores()` 只先恢复当前页面所需状态，技能扫描和 MCP 自动重连必须留在后台慢任务里。
31. 外观相关的首屏读取与应用逻辑优先收敛在独立 helper 中，不要继续把主题/背景 bootstrap 细节堆回 `stores/index.ts`。

## 4. 当前后端边界

- 前端负责：聊天界面、Skills 展示、MCP 配置、任务工作台、会话持久化。
- 后端负责：模型调用桥接、MCP 连接与工具调用、后续托管任务、即时 Skill 执行。
- 当前已完成：
  - 聊天走后端 `/api/chat`
  - 流式聊天走后端 `/api/chat/stream`
  - 模型列表走后端 `/api/models`
  - MCP 已进入 `server/`
  - 托管任务第一版执行器已进入 `server/`
  - 即时 Skill 第一版执行链已进入 `server/`
- 上游 TLS 证书链异常优先在 `server/` 兜底，不要把这类兼容逻辑塞回前端
- 这类兜底改动要保留原始错误 `cause`，不要在中间层把诊断信息包丢
- 识别网络/TLS 错误时要允许沿 `cause` 链向下追，不要只看最外层异常
- 外观状态的首屏 bootstrap 要和 store 初始状态保持一致；背景颜色预设属于浅色外观的一部分，切换预设时不要留下“底板已切、组件变量未切”的半状态。

## 5. 当前已知重点问题

- `src/components/Chat/InputArea.tsx` 仍然偏重。
- `src/stores/index.ts` 仍然偏大。
- `src/components/panels/ToolsPanel.tsx` 后续应拆成独立 tab 组件。
- `src/components/Scripts/ScriptsWorkspace.tsx` 后续也值得拆成列表区 / 详情区 / 托管表单子组件。
- `scripts/` 与 `skills/` 的职责边界还要继续收紧。
- Skill 文件保持最小可用集：`skill.yaml` 只保留执行需要的字段，`instructions.md` 保持短说明，不再堆重复教程。
- Skill 元数据透传时不要丢 `defaultArgs`，否则托管任务和即时任务都会出现“UI 看起来能跑，实际参数为空”的问题。
- 托管任务进程退出后要及时清空运行句柄，不能把已退出进程继续当作活跃任务缓存。
- 托管任务一旦进入 `stopping`，日志解析不要再把状态覆盖回 `warning/error`，否则 UI 会出现“已停但仍是异常态”的假象。
- `stdio` MCP 出问题时，后端必须给出超时或 `stderr` 线索，不能长期挂住不返回。
- 聊天中的 MCP 规划只接受真实 tool call；像 `<invoke>`、`<parameter>` 这类伪工具标记必须拦截，不能直接展示给用户。
- 聊天侧 MCP 采用“双层规划”：先尝试模型原生 tool call；如果模型没有稳定给出真实调用，再用本地确定性规划兜底。
- MCP 对话不能只做单轮工具规划；如果模型先用一个只读工具探路，后续仍要允许继续调用真实工具，直到拿到足够结果或明确失败。
- `filesystem` 这类内置 MCP 的默认风险分级要让读操作保持 `read-only`，不要把 `read_text_file`、`directory_tree` 之类读工具误标成高风险，否则会直接影响聊天侧的工具选择。
- MCP 多轮对话里，模型第二轮可能直接返回 `server/tool` 形式的真实工具名，不能只识别首轮注入的别名。
- 总览页任务卡中的状态标签要使用专用选择器，不能再被宽泛的 `span` 规则误伤，否则会出现 pill 尺寸和排版畸变。
- “运行中 / 已恢复” 在任务卡和总览里统一按绿色健康态展示，不要再和中性灰态混用。
- 当前工作区和当前对话这类 UI 快照要优先写本地缓存并在启动时优先恢复，避免刷新时短暂掉回默认首页或空白初始化态。
- 启动初始化要区分“首屏关键恢复”和“后台慢任务”；技能扫描、MCP 自动重连这类慢步骤不要阻塞首屏渲染。
- 文件类 MCP 请求必须先补全 `path` 再发起调用；不能把 bare filename 的未解析状态直接传给后端。
- 默认 `filesystem` MCP 根目录应收敛到当前项目根，不能继续使用 `/Users` 这类过宽目录，否则聊天规划会被错误目录污染。
- 对 `filesystem` 这类文件型 MCP，请求一旦命中明确文件/目录目标，应优先走本地确定性规划，再把结果交回模型总结，不要先让模型自由编目录。

## 6. MCP 当前约定

- 当前后端已支持：
  - `streamable-http`
  - `stdio`（最小可用）
- `stdio` 相关增强应继续优先落到 `server/`，不要把进程治理逻辑塞回前端。
- MCP 新改动优先进入 `server/src/index.js`，不要把协议细节塞回前端组件。

## 7. 仓库清理约束

不要提交这些内容：

- `dist/`
- `node_modules/`
- 未使用 stub
- 默认模板资源
- 编辑器私货

## 8. 一句话原则

继续按“前端展示 + runtime 抽象 + server 承接能力”的方向演进。

- 首屏恢复优先使用同步 bootstrap 快照；不要再用额外的 hydration 标志把聊天区挂成空白壳。
