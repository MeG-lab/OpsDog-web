# AIops 用户使用说明

## 1. 文档说明

本文档面向 AIops 的实际使用者，重点说明以下内容：

- 如何开始使用 AIops
- 如何配置大模型
- 如何配置 Python 环境
- 如何使用即时任务与托管任务
- 如何编写和配置 Skill
- 如何组织 Python 脚本目录
- 如何查看系统通告
- 如何排查常见问题

本文档以当前项目版本为基准，适用于内部演示、试用和日常使用。

---

## 2. AIops 是什么

AIops 是一个面向运维场景的桌面端 AI 工作台。

它的核心理念是：

- **对话即需求**
- **脚本即功能**

你可以把 AIops 理解为一个统一的运维入口：

- 用户通过对话描述需求
- 系统理解需求并匹配 Skill
- 对应脚本被执行或托管运行
- 状态、日志和系统事件统一展示

当前 AIops 主要包含 4 类能力：

1. **对话问答**
2. **即时任务**
3. **托管任务**
4. **系统通告**

## 5. Python 环境配置说明

## 5.1 Python 在 AIops 中的作用

Python 是 AIops 执行脚本能力的核心运行环境。

当前以下能力依赖 Python：

- 即时任务执行
- 托管任务执行
- Skill 对应入口脚本执行

所以，正确配置 Python 解释器非常重要。

---

## 5.2 配置位置

进入 **设置** 后，可以看到 Python 路径相关配置项。

你需要填写的是：

**Python 可执行文件路径**

例如：

```bash
python3
```

或者：

```bash
/usr/bin/python3
```

或者：

```bash
/opt/homebrew/bin/python3
```

---

## 5.3 推荐配置方式

如果系统环境已经正确安装 Python，通常可以直接填写：

```bash
python3
```

如果你的电脑上有多个 Python 版本，建议填写绝对路径。

这样可以避免：

- 调用到错误的解释器
- 缺少依赖
- 执行脚本失败

---

## 5.4 如何确认 Python 可用

你可以在终端运行：

```bash
python3 --version
```

如果能正常看到版本号，说明 Python 基本可用。

如果 AIops 中脚本仍然无法执行，请进一步确认：

- 路径是否和终端里一致
- 当前 Python 是否装有对应依赖

---

## 5.5 Python 相关常见问题

### 1. 脚本提示找不到 Python

请检查：

- Python 路径是否填写正确
- 系统环境变量是否可用

### 2. 脚本能启动但报依赖缺失

说明 Python 本身可用，但当前解释器没有安装脚本需要的依赖。

需要在对应解释器环境中安装依赖。

## 6. 脚本目录说明

## 6.1 当前目录结构

AIops 的脚本目录分为两套语义：

- **项目目录**
  用于开发阶段、自带示例脚本和调试
- **用户目录**
  用于打包后的正式使用、用户上传和后续维护

正式使用时，优先使用 **用户目录**。

---

## 6.2 打包版脚本上传位置

打包成 `.app` 或 `.dmg` 后，**不要把 Python 文件上传到应用包内部**。

正确的上传位置是：

### 即时任务脚本目录

```bash
~/.aiops/scripts/instant/
```

### 托管任务脚本目录

```bash
~/.aiops/scripts/managed/
```

原因是：

- 应用包内部资源不适合运行时增删
- 应用升级后，包内文件可能被覆盖
- 用户脚本、托管脚本、后续扩展都更适合放在用户目录

AIops 当前已经按这个优先级解析脚本：

1. 先找 `~/.aiops/scripts/`
2. 再找 Skill 自身目录
3. 最后才回退到项目目录（开发态兜底）

---

## 6.3 项目目录结构

项目开发阶段，当前脚本按任务类型分为两类目录：

### 即时任务脚本目录

```bash
scripts/instant/
```

用于存放一次性执行的脚本，例如：

- ping 测试
- 巡检
- 日志分析
- 临时诊断

### 托管任务脚本目录

```bash
scripts/managed/
```

用于存放持续运行脚本，例如：

- 端口监控
- 进程守护
- 日志守护
- 长期轮询任务

---

## 6.4 当前示例脚本

### 即时任务脚本

```bash
scripts/instant/test_ping.py
scripts/instant/ping.txt
```

### 托管任务脚本

```bash
scripts/managed/service_watchdog.py
scripts/managed/monitor.py
```

---

## 6.5 目录使用建议

### 即时任务

新的一次性脚本建议放到：

```bash
~/.aiops/scripts/instant/
```

### 托管任务

新的持续运行脚本建议放到：

```bash
~/.aiops/scripts/managed/
```

开发阶段如果需要随仓库一起维护示例，也可以保留在：

```bash
scripts/managed/
```

这样后续 Skill 配置、任务分类和产品展示都会更清晰。

## 7. Skill 配置说明

## 7.1 Skill 是什么

Skill 是 AIops 中的能力描述文件。

它的作用是：

- 告诉系统这个能力叫什么
- 告诉系统它是即时任务还是托管任务
- 告诉系统入口脚本在哪里
- 告诉系统它适合处理什么需求

可以把 Skill 理解成：

**脚本的产品化配置层**

---

## 7.2 Skill 目录结构

一个 Skill 通常至少包含：

```bash
skills/<skill_name>/
  skill.yaml
  instructions.md
```

其中：

- `skill.yaml`：元数据配置
- `instructions.md`：给模型看的使用说明

---

## 7.3 Skill 扫描目录

当前系统会扫描两个位置：

### 项目内置 Skills

```bash
skills/
```

### 用户目录 Skills

```bash
~/.aiops/skills/
```

也就是说，内置 Skill 和上传 Skill 都能被系统识别。

---

## 7.4 skill.yaml 基本字段说明

一个典型的 `skill.yaml` 例如：

```yaml
name: service_watchdog
version: 1.0.0
description: 持续检测本地 7001 端口的可用性，适合作为托管任务能力验证
triggers:
  - "托管端口检测"
  - "持续检测服务状态"
  - "检查7001端口"
  - "service_watchdog"
task_kind: managed
entry_script: scripts/managed/service_watchdog.py
timeout_seconds: 0
dependencies: []
default_args:
  - "--host"
  - "127.0.0.1"
  - "--port"
  - "7001"
  - "--interval"
  - "3"
```

---

## 7.5 重点字段解释

### `name`

Skill 名称，建议唯一。

例如：

```yaml
name: ping_checker
```

### `description`

Skill 说明，用于展示和理解用途。

### `triggers`

Skill 标签和触发词，用于帮助系统匹配用户需求。

例如：

```yaml
triggers:
  - "ping检测"
  - "网络连通性"
```

### `task_kind`

表示任务类型，目前支持：

```yaml
instant
managed
```

- `instant`：即时任务
- `managed`：托管任务

### `entry_script`

入口脚本路径。

推荐保持相对路径写法，例如：

```yaml
entry_script: scripts/instant/test_ping.py
```

或者：

```yaml
entry_script: scripts/managed/service_watchdog.py
```

运行时解析规则是：

1. 优先解析到 `~/.aiops/scripts/...`
2. 如果 Skill 自带脚本，再解析 Skill 目录
3. 最后开发态回退到项目目录

所以打包版正式使用时，只需要把实际 Python 文件放到用户目录即可。

### `timeout_seconds`

脚本超时时间。

即时任务可以设置为有限秒数，托管任务通常可设为：

```yaml
timeout_seconds: 0
```

表示不按一次性脚本超时退出。

### `default_args`

默认参数，用于给托管任务或脚本提供初始配置。

---

## 7.6 instructions.md 的作用

`instructions.md` 主要用于给模型提供“这个 Skill 是干什么的、应该怎么理解”的说明。

注意：

- 它不是脚本本体
- 它主要是帮助模型理解该 Skill 的用途
- 它可以说明脚本在哪里、如何使用、适合什么场景

---

## 7.7 Skill 上传与删除

当前 AIops 已支持在客户端中：

- 上传 Skill

- 编辑说明
- 编辑标签

也就是说，不一定需要手动复制文件到目录，普通使用者可以直接在客户端中完成管理。

## 8. 即时任务使用说明

## 8.1 即时任务是什么

即时任务用于一次性执行场景，例如：

- ping 检测
- 日志分析
- 巡检
- 临时诊断

它的特点是：

- 执行一次
- 返回结果
- 不持续运行

---

## 8.2 当前示例：ping_checker

当前内置了一个示例即时任务：

```bash
ping_checker
```

它会调用：

```bash
scripts/instant/test_ping.py
```

并读取：

```bash
scripts/instant/ping.txt
```

对其中的目标进行连通性检测。

---

## 8.3 如何触发即时任务

可以通过对话触发，例如：

```text
帮我做一次 ping 测试
```

或者：

```text
检查网络是否正常
```

如果命中 Skill，系统会执行对应脚本并把结果回填到对话中。

---

## 9. 托管任务使用说明

## 9.1 托管任务是什么

托管任务用于持续运行场景，例如：

- 持续监控端口
- 持续监控进程
- 长时间日志守护
- 周期性检测任务

它的特点是：

- 长时间运行
- 持续输出状态
- 支持异常与恢复
- 支持启动 / 停止 / 重启

---

## 9.2 当前示例：service_watchdog

当前托管任务示例为：

```bash
service_watchdog
```

入口脚本：

```bash
scripts/managed/service_watchdog.py
```

当前主要用于：

- 检测某个端口是否可连接
- 周期性检查
- 记录运行、异常、恢复状态

---

## 9.3 托管任务配置项

当前在脚本详情页中，可配置以下参数：

- 主机
- 端口
- 间隔（秒）
- 连续失败阈值
- 日志文件

这些配置当前已经支持持久化保存。

---

## 9.4 托管任务操作

当前支持：

- 启动托管
- 停止托管
- 重启托管

并支持查看：

- 当前状态
- 最近日志
- 最近异常
- 最近恢复

---

## 9.5 托管任务状态说明

当前常见状态包括：

- `运行中`
- `需关注`
- `告警中`
- `已恢复`
- `停止中`
- `已停止`
- `异常退出`

---

## 10. 系统通告使用说明

## 10.1 系统通告是什么

系统通告是独立的系统消息流，用于承载：

- 托管任务异常
- 托管任务恢复
- 后续可扩展的系统级事件

它不是普通聊天窗口，而是：

**只读通知中心**

---

## 10.2 当前支持功能

系统通告当前支持：

- 固定置顶
- 未读数量提示
- 清空通告
- 只读消息流

当前打开系统通告时，不会显示输入框。

---

## 10.3 为什么要单独做系统通告

原因是：

- 系统异常不应该污染业务对话
- 恢复通知也不应该混到普通聊天里
- 所有系统级事件应该有独立入口

---

## 11. MCP 使用说明

## 11.1 MCP 在 AIops 中的定位

MCP 在当前项目中是：

**扩展能力接入层**

它不是核心执行闭环，但可以帮助系统接入更多外部工具和外部数据能力。

---

## 11.2 当前支持内容

当前 MCP 已支持：

- 配置 MCP Server
- 连接 / 断开
- 自动重连已启用服务
- 工具列表获取
- 对话中调用 MCP 工具

---

## 11.3 当前使用建议

目前更适合将 MCP 理解为：

- 外部工具集成
- 文件系统能力补充
- 未来接企业系统的外部入口

它不应该替代 Skill 和 Python 脚本的核心能力。

## 13. 托管脚本输出规范

## 13.1 为什么要统一格式

托管任务脚本建议输出标准 JSON 行，以便：

- 前端统一展示
- 对话查询统一解析
- 异常通知统一识别
- 后续报告统一生成

---

## 13.2 推荐最小结构

建议托管任务每一行输出一条 JSON，至少包含：

```json
{
  "time": "2026-04-25T00:11:28",
  "level": "running",
  "message": "service healthy"
}
```

推荐附加字段：

```json
{
  "target": {
    "host": "127.0.0.1",
    "port": 7001
  },
  "details": ["tcp connect ok: 127.0.0.1:7001"],
  "consecutiveFailures": 0
}
```

---

## 13.3 常见 level 建议

建议统一使用：

- `info`
- `running`
- `attention`
- `warning`
- `recovered`
- `error`

## 14. 常见问题

## 14.1 为什么 Skill 看不到

请检查：

- Skill 是否放在 `skills/` 或 `~/.aiops/skills/`
- `skill.yaml` 是否格式正确
- 是否已在客户端中启用

---

## 14.2 为什么脚本无法执行

请检查：

- Python 路径是否正确
- 入口脚本是否存在
- `entry_script` 配置是否正确
- 脚本依赖是否安装完成

---

## 14.3 为什么托管任务没有状态变化

请检查：

- 监控目标是否正确
- 脚本是否持续输出 JSON
- 任务是否真的处于运行状态

---

## 14.4 为什么系统通告没有消息

请检查：

- 托管任务是否真的发生了异常或恢复
- 是否只是一直处于正常状态
- 当前是否已经看过该通知并变成已读

---

## 14.5 为什么系统通告不能输入内容

这是设计如此。

系统通告是：

- 系统事件流
- 只读消息通道

不是普通聊天窗口。

---
