# AIops 产品整体报告

## 1. 产品概述

### 1.1 产品名称

**AIops 智能运维助手**

### 1.2 一句话定义

AIops 是一个面向运维、平台工程、研发支持和内部技术团队的 **桌面端 AI 运维工作台**。  
它以“**对话即需求，脚本即功能**”为核心理念，把自然语言交互、运维脚本、持续监控、工具集成和系统通知整合到一个统一的工作环境中。

### 1.3 产品定位

与传统“聊天机器人”不同，AIops 不是只负责回答问题的问答助手，而是一个真正可执行、可配置、可持续运行的 **运维智能体工作台**。

它主要解决三类问题：

1. 运维人员需要频繁在终端、脚本、日志、监控、知识和聊天之间来回切换，工作链路割裂。
2. 很多脚本能力长期存在，但分散在文件夹、个人电脑、Wiki、群聊经验中，难以沉淀为团队共享能力。
3. 运维工作里既有“查一次”的即时任务，也有“持续盯着”的托管任务，传统工具往往只能覆盖其中一部分。

AIops 的目标，就是把这些能力统一到一个产品内：

- 用户通过对话描述需求
- 系统匹配合适的技能（Skills）和脚本
- 即时任务可以直接执行
- 托管任务可以持续运行、持续检测、持续反馈
- 系统异常和恢复可以进入统一的系统通知流

## 2. 产品要解决的核心问题

### 2.1 运维工作链路碎片化

在实际企业环境中，运维工作常见的操作路径是：

- 用户或业务提出问题
- 运维同学打开终端
- 找脚本
- 改参数
- 看日志
- 再回群里回复
- 再切到监控系统确认状态

这个过程中，信息和操作分散在多个地方，效率低，且难以沉淀。

AIops 把这些动作抽象成统一产品能力：

- **对话**：用户入口
- **脚本**：执行能力
- **托管任务**：持续监控能力
- **系统通告**：系统级事件入口
- **MCP / Skills**：外部与内部能力扩展层

### 2.2 脚本资产没有产品化

很多企业内部都有大量 Python、Shell、日志分析、巡检、诊断脚本，但往往存在以下问题：

- 只有少数人知道脚本在哪
- 参数和用法靠口口相传
- 无法用统一方式管理、上传、启停、分类
- 没有明确区分“一次性脚本”和“持续运行脚本”

AIops 通过 Skills 和脚本工作台，把脚本从“文件”提升为“功能”。

### 2.3 缺少对运维场景友好的 AI 交互

通用大模型擅长回答，但不天然擅长：

- 理解企业内部脚本体系
- 执行本地任务
- 持续监控服务状态
- 生成系统级告警通知
- 承载可持续运行的运维任务

AIops 在此基础上补上了一层“运维产品能力”，把模型从聊天入口延伸到执行层、监控层和通知层。

## 3. 产品核心理念

### 3.1 对话即需求

用户不需要先思考“我要点哪个菜单”“我要进哪个系统”，而是直接说：

- 帮我做一次 ping 测试
- 7001 端口最近怎么样
- 帮我持续监控这个端口
- 查看托管任务状态

系统负责理解这句话代表的是：

- 普通问答
- 即时任务执行
- 托管任务查询
- 托管任务创建建议

### 3.2 脚本即功能

AIops 不把脚本视为技术文件，而把它们视为产品功能单元。

一个脚本一旦通过 Skill 进行元数据包装，就拥有了：

- 名称
- 类型（即时任务 / 托管任务）
- 说明
- 标签 / 触发词
- 入口脚本
- 默认参数

这意味着脚本不再只是“某个目录里的某个 py 文件”，而是一个可被对话调用、可被 UI 展示、可被团队管理的能力模块。

### 3.3 持续任务不是监控图表，而是可托管的智能能力

很多产品的“监控”只提供图表和告警。  
AIops 的设计更偏“任务视角”：

- 可以启动一个持续运行脚本
- 可以停止、重启
- 可以看到当前状态和最近日志
- 可以在系统通告中看到异常与恢复

这意味着 AIops 更像是一个“值守智能体平台”，而不只是数据展示面板。

## 4. 当前产品整体能力

## 4.1 大模型接入能力

当前产品已经支持多类模型接入，并完成了可用的配置和对话闭环。

支持的模型提供商包括：

- OpenAI 兼容
- Anthropic Claude
- Google Gemini
- 阿里百炼
- DeepSeek
- 硅基流动
- 火山方舟
- 智谱 AI
- 月之暗面
- 自定义兼容服务

已实现能力包括：

- 配置 API Key
- 配置 Base URL
- 一键拉取模型列表
- 选择模型并持久化保存
- 直接开始对话

### 4.2 Skills 体系

Skills 是 AIops 的能力描述层，用于把脚本、说明和使用方式包装成系统可识别的“技能”。

当前已经支持：

- 扫描项目内 `skills/` 与用户目录 `~/.aiops/skills/`
- 在客户端中上传 Skill
- 删除 Skill
- 编辑 Skill 说明和标签
- 启用 / 停用 Skill
- 将 Skill 注入对话能力链

已验证的 Skill 包括：

- `ping_checker`
- `service_watchdog`

### 4.3 即时任务

即时任务用于一次性执行的运维能力，例如：

- 网络连通性检测
- 巡检
- 日志分析
- 临时诊断
- 报告生成

当前已实现：

- Skill 匹配
- 本地脚本调用
- Python 执行链
- 结果回填对话
- 脚本工作台中按“即时任务”分类展示

### 4.4 托管任务

托管任务是当前产品最重要的差异化能力之一，代表“持续运行、持续检测、持续反馈”的任务模型。

当前已实现：

- 托管脚本启动
- 托管脚本停止
- 托管脚本重启
- 托管参数配置表单
- 参数配置持久化
- 当前状态展示
- 最近日志展示
- 对话查询托管状态
- 顶部托管摘要
- 异常 / 恢复自动进入系统通告

已验证脚本：

- `service_watchdog.py`

它支持：

- 检查端口存活
- 周期性轮询
- 连续失败升级
- 恢复检测
- 优雅停止
- 标准 JSON 输出

### 4.5 MCP 工具集成

MCP 在本项目中的定位是 **扩展能力接入层**，不是核心闭环依赖层。

当前已实现：

- MCP Server 配置
- 连接 / 断开
- 工具列表获取
- 配置持久化
- 自动重连已启用服务
- 聊天中调用 MCP 工具

产品定位上：

- 核心闭环仍然是对话 + Skill + Python 脚本 + 托管任务
- MCP 用于补充外部系统能力

### 4.6 系统通告

系统通告是产品中的系统级消息通道，用于承载：

- 托管任务异常
- 托管任务恢复
- 后续可扩展的 MCP 异常、Skill 失败、自动化提醒

当前已实现：

- 固定置顶系统会话
- 未读数量提示
- 清空系统通告
- 只读消息流
- 启动时不抢占首页
- 与普通业务对话解耦

## 6. 产品特色

### 6.1 面向运维，不是面向泛助手

AIops 是一个 **运维工作产品**，不是简单的通用 AI 聊天壳。

它的产品结构天然围绕运维场景展开：

- 即时任务
- 托管任务
- 系统通告
- 脚本管理
- 技能管理

### 6.2 把“脚本能力”做成了“产品功能”

很多同类产品只是在聊天里调用 shell。  
AIops 则进一步把脚本做成可见、可配、可托管、可通知的功能单元。

### 6.3 同时覆盖“执行”和“值守”

传统 AI 助手通常只覆盖“执行一次”的任务；  
AIops 同时覆盖：

- 即时执行
- 持续运行
- 状态追踪
- 系统通知

这让它更适合真实的企业运维场景。

### 6.4 更容易沉淀团队内部能力

通过 Skill 和脚本工作台，团队内部脚本资产可以统一沉淀，而不是继续依赖个人经验和私有目录。

### 6.5 更适合作为企业内部运维入口

AIops 很适合做成：

- 内部运维助手
- 平台工程工具台
- 团队脚本管理入口
- 值守监控与诊断入口

- 对话界面

- 设置 / 工具集成

#### 第二层：能力调度层

- Skill 匹配
- 对话意图识别
- 即时任务执行调度
- 托管任务状态查询与控制

#### 第三层：执行层

- Python 脚本执行

#### 第四层：外部连接层

- LLM Provider
- MCP Servers
- 本地文件系统
- 后续企业系统（OA、工单、CMDB、审批流、知识库等）

## 8. 与 OpenClaw 的对比分析

> 本节基于 OpenClaw 官方 GitHub、官网和文档公开信息整理，截至 2026-04-25。  
> 参考来源：  
> [OpenClaw GitHub](https://github.com/openclaw/openclaw)  
> [OpenClaw 官网](https://openclaw.ai/)  
> [OpenClaw Docs](https://docs.openclaw.ai/index)

### 8.1 OpenClaw 的定位

从官方信息看，OpenClaw 更偏向：

- 个人 AI 助手
- 跨聊天渠道代理
- 多渠道接入
- 个人生产力自动化
- 多终端、多平台消息驱动

它的核心优势在于：

- WhatsApp / Telegram / Slack / Discord / iMessage 等多渠道接入
- 个人助手式常驻服务
- 丰富的外部集成能力
- 强调“任意聊天渠道驱动 AI 助手”

### 8.2 AIops 的定位

AIops 与 OpenClaw 的核心差异在于：

AIops 不是“跨聊天渠道的个人助理”，而是 **面向企业运维和平台工程场景的工作台产品**。

它更强调：

- 运维脚本沉淀
- 运维任务分类
- 托管任务持续运行
- 状态追踪和告警
- 系统级消息流
- 面向企业内部的能力治理

### 8.3 AIops 相比 OpenClaw 的优势

#### 优势 1：更聚焦运维场景

OpenClaw 的应用范围更广，但也更泛。  
AIops 则天然围绕运维场景设计，因此在以下方面更直接：

- 即时任务与托管任务区分明确
- 系统通知中心更符合运维语义
- Python 脚本资产更易沉淀为团队能力
- 更适合作为内部运维入口

#### 优势 2：脚本产品化程度更高

OpenClaw 擅长技能与工具扩展，但其核心仍是“个人助手”和“多渠道 AI 代理”。  
AIops 在本项目中把脚本提升到了产品一级能力，具备：

- 功能视图
- 任务分类
- 参数配置
- 持续运行
- 告警通知

这一点对企业运维团队尤其重要。

#### 优势 3：托管任务模型更贴近值守场景

AIops 中的托管任务并不是“定时自动化动作”那么简单，而是带有：

- 持续运行
- 持续状态检测
- 异常与恢复识别
- 系统通知流

这比单纯的 Agent 调度更适合运维团队的长期值守逻辑。

#### 优势 4：产品边界更清晰

OpenClaw 是强大的个人 AI 助手平台，但功能广度也意味着企业化运维产品边界不一定天然清晰。  
AIops 则天然是：

- AI 运维工作台
- 内部运维助手
- 脚本与技能管理平台

在企业内部推广时，定位会更直接。

### 8.4 OpenClaw 相比 AIops 的优势

为了客观说明，OpenClaw 也有显著优势：

- 多渠道接入能力远强于当前 AIops
- 生态规模和开源社区更成熟
- 个人助手、消息入口、移动节点方面更完整
- 在跨聊天平台和个人生产力自动化方面积累更深

### 8.5 两者关系的最佳理解

OpenClaw 更像：

**跨平台个人 AI 助手操作系统**

AIops 更像：

**企业内部 AI 运维工作台**

如果企业要做运维场景落地，AIops 更容易贴近内部工作流程；  
如果目标是做跨聊天平台的个人 AI 助手生态，OpenClaw 的优势更明显。

这意味着 AIops 不是一个 Demo，而是一个具备继续演化价值的企业内部产品底座。
