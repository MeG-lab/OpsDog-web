# AIops智能运维中枢 风险与已知限制清单

## 1. 文档目的

本文档用于说明 AIops智能运维中枢 当前版本在交付阶段的已知风险、边界和限制。  
本清单不是“缺陷列表”，而是帮助使用方、验收方和后续维护者理解当前版本的真实边界。

---

## 2. 当前版本定位

当前版本可定位为：

**企业内部一期可交付版本**

具备：

- 对话入口
- 即时任务执行
- 托管任务值守
- 系统通知
- MCP 外部能力接入
- 审计记录与查询

但仍属于：

**企业级平台的第一阶段版本**

因此在“规模化、守护强度、治理深度、流程集成”方面仍有继续增强空间。

---

## 3. 已知技术限制

### 3.1 托管任务还不是系统级服务守护

当前托管任务已经支持：

- 后台独立运行
- 状态快照持久化
- PID 重连
- 受控自动恢复

但仍未做到：

- 系统服务级 daemon / sidecar 守护
- 脱离应用后的完整生命周期管理
- 系统启动即自动随 OS 恢复

影响：

- 对大规模、长期值守场景仍不是最终形态
- 当前更适合内部平台试点和一期交付，而不是高强度生产级守护系统

建议：

后续引入独立 sidecar 或系统服务托管模型。

---

### 3.2 审计中心仍为轻量版

当前审计能力已支持：

- 日志落盘
- 查询
- 过滤
- 搜索
- 回放

但仍未做到：

- 时间范围筛选
- 风险等级可视化
- 审计导出
- 审计归档策略
- 大规模事件分页加载

影响：

- 适合一期治理与问题追踪
- 还不适合作为大型企业合规审计中心的最终版本

---

### 3.3 聊天数据虽然进入 SQLite，但前端仍保留整批保存兜底

当前版本已经：

- 使用 SQLite 作为主存储
- 支持会话表与消息表
- 提供细粒度 CRUD 接口

但前端聊天链仍保留整批 `saveConversations` 兜底机制。

影响：

- 当前可靠性已经足够交付
- 但在极大规模会话量和更复杂并发写入场景下，仍建议进一步纯增量化

---

### 3.4 MCP 风险控制已建立，但治理深度仍有限

当前已支持：

- 显式请求才注入
- 风险级别裁剪
- 工具级风险标签
- 结构化确认

但仍未做到：

- 更复杂的组织级权限策略
- 用户/角色维度的 MCP 工具隔离
- 工具级审批流

影响：

- 当前适合本地桌面单用户场景
- 若扩展为多人、多角色或远程平台，需要继续加强权限体系

---

### 3.5 Skill / 脚本生态仍依赖规范维护

当前系统已经支持：

- Skill 元信息
- 参数 schema
- 脚本目录规范
- 托管脚本 JSON 输出规范

但仍存在：

- 新增脚本和 Skill 需要遵循规范
- 若脚本不按规范实现，前端展示与对话理解质量会下降

影响：

- 当前平台适合“受控扩展”
- 还不适合完全无规范地开放给大量第三方脚本

---

## 4. 当前版本的安全边界

### 4.1 当前已做的安全收口

当前版本已经完成：

- 后端意图路由
- 高风险输入阻断
- 提示注入识别
- MCP 显式请求门槛
- MCP 风险级别裁剪
- Skill 参数 schema 校验
- API Key 系统凭据库存储
- 路由/执行/工具调用审计

因此当前安全边界已经明显强于早期原型版本。

### 4.2 当前仍需明确的边界

当前版本仍然默认是：

- 本地桌面端
- 单用户
- 企业内部受控环境

不应直接理解为：

- 公网暴露服务
- 多租户 SaaS
- 多人共享控制平台

如果未来往这些方向扩展，必须追加：

- 认证
- 授权
- 用户隔离
- 远程访问安全策略

---

## 5. 使用与交付风险

### 5.1 Python 环境依赖风险

脚本执行依赖本机 Python 环境。  
如果目标机器：

- Python 路径错误
- 依赖缺失
- 权限不足

会影响脚本能力。

建议：

- 交付前在目标机器验证 Python 路径
- 尽量明确内置或推荐版本
- 提供标准依赖安装说明

---

### 5.2 用户目录权限风险

当前平台依赖：

- `~/.aiops/config.json`
- `~/.aiops/data.db`
- `~/.aiops/skills/`
- `~/.aiops/scripts/`
- `~/.aiops/managed_tasks.json`
- `~/.aiops/managed_logs/`

如果用户目录无写权限或被系统策略限制，会影响配置、会话、托管与日志功能。

---

### 5.3 网络与模型服务可用性风险

LLM 依赖外部服务。  
如果模型服务不可用，会影响：

- 普通对话
- Skill 说明增强
- 对话中的辅助分析

但本地：

- 托管任务
- 即时脚本执行
- 系统通知

仍可继续工作。

建议：

在交付说明中明确区分：

- 模型依赖能力
- 本地确定性能力

---

## 6. 当前版本不建议承诺的能力

交付时不建议对外承诺以下内容已经完全成熟：

1. 全自动高危运维动作执行
2. 多角色权限隔离
3. 系统级服务守护
4. 大规模审计治理中心
5. 完整 OA / 工单 / 工作流集成
6. 报告导出与正式交付件自动流转

这些可以作为后续版本规划，但不应在当前版本验收口径中当作“已完全实现”。

---

## 7. 当前版本适合的交付范围

当前版本更适合：

1. 企业内部试点
2. 运维团队/平台团队工具交付
3. 部门级智能运维工作台
4. 本地值守、脚本调度、托管监控、系统通知、审计留痕的一期平台

---

## 8. 后续优先增强建议

建议后续优先推进：

1. 托管任务独立守护模型进一步升级
2. 聊天链彻底纯增量化
3. 审计中心继续产品化
4. 报告导出与交付能力
5. OA / 工单 / 工作流集成

---

## 9. 结论

当前版本已经具备：

**可初步交付、可内部使用、可继续平台化演进** 的条件。

它的主要风险不在于“是否能用”，而在于：

- 还不是最终形态
- 还存在平台化与治理深度上的继续升级空间

因此，当前最合适的交付话术应是：

**AIops智能运维中枢 当前版本已达到企业内部一期交付标准，具备正式试点和部门级落地条件。**
