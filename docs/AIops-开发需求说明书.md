# AIops 开发需求说明书

## 1. 项目名称

**AIops 智能运维助手**

---

## 2. 项目目标

AIops 的目标是打造一个面向企业内部运维场景的桌面端智能工作台，实现：

- 通过自然语言完成运维需求表达
- 将脚本能力沉淀为可管理、可调用、可持续运行的功能模块
- 支持即时任务执行与托管任务持续运行
- 支持系统级异常通知、状态追踪与任务交付
- 逐步从“对话及需求，脚本及功能”演进到“对话及交付，脚本及功能”

---

## 3. 产品定位

AIops 不是通用聊天机器人，而是一个 **AI 运维工作台**。

产品核心定位：

- **对话**：用户入口
- **脚本**：执行能力
- **Skill**：能力描述与能力管理
- **托管任务**：持续运行与持续监控能力
- **系统通告**：系统级事件流
- **MCP**：扩展工具接入层
- **报告输出**：后续面向交付和汇报的重要能力

---

## 4. 当前产品理念

### 4.1 第一层理念

**对话即需求，脚本即功能**

即：

- 用户通过对话描述需求
- 系统理解需求并匹配对应脚本或能力
- 脚本不再只是文件，而是功能单元

### 4.2 第二层理念

**对话及交付，脚本及功能**

即：

- 不仅要能回答和执行
- 还要能持续运行、产生通知、形成交付件
- 后续逐步扩展到报告、流程、OA、工单、知识库等场景

---

## 5. 总体开发范围

本项目整体开发范围包括以下几个部分：

1. 大模型接入与聊天能力
2. Skills 体系
3. 即时任务体系
4. 托管任务体系
5. MCP 工具集成
6. 系统通告体系
7. UI 与交互体验

9. 企业内部系统扩展能力

### 第一阶段：可用

1. 大模型接入
2. 聊天基础能力
3. Skills 扫描与管理
4. 即时任务执行
5. 托管任务配置、启停、重启

### 第二阶段：好用

1. 对话查询增强

1. 对话识别托管创建意图
2. 对话给出结构化创建建议
3. 异常与恢复自动写入系统通告

### 第四阶段：输出与交付

1. 结构化报告数据模型
2. 巡检报告模板
3. 故障分析报告模板

5. Word 输出

### 第五阶段：企业扩展

1. OA 接入
2. 工作流接入
3. 工单系统接入
4. CMDB / 资产系统接入
5. 企业知识库接入

## 9. 后续重点开发方向

当前产品基础能力已基本形成，后续重点方向建议如下：

### 9.1 报告与交付能力

这是产品从“回答与执行”走向“输出与交付”的关键一步。

建议顺序：

1. 定义报告结构
2. 生成巡检与诊断报告
3. 输出 PDF
4. 再补 Word

### 9.2 对话直接创建并启动托管任务

目前已能识别托管创建意图并给出建议，下一步应实现：

- 从对话直接创建托管任务
- 直接写入配置
- 直接启动任务

### 9.3 更多托管模板

逐步补齐：

- 端口监控
- 进程守护
- 日志守护
- 磁盘阈值监控
- HTTP 健康检查

### 9.4 企业内部系统接入

后续适合对接：

- 企业 OA
- 工作流
- 工单系统
- 知识库
- 监控与可观测平台

## 10. 演示重点建议

如果该说明书用于演示，建议重点讲以下内容：

1. **AIops 是一个 AI 运维工作台，不只是聊天工具**
2. **对话即需求，脚本即功能**
3. **即时任务 + 托管任务** 是产品核心差异点

- 对话入口
- 脚本执行
